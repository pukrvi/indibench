The open, lab-neutral frontier benchmark for India

IndiBench evaluates LLMs and agents on Indian languages and Indian use cases — across text, speech, vision, and agentic tasks — at a difficulty where today's best models should score low. That is the point.

The problem

Frontier AI has a benchmark suite it respects — Humanity's Last Exam for expert knowledge, SWE-bench for software engineering. India is absent from that suite. Every open Indian benchmark today is one of:

Saturated

Frontier models already score 56–77% on MILU, BhashaBench, ParamBench. They measure deployment competence, not frontier progress.

Contaminated

Harvested from public exam papers that are already all over the web — so they leak into training corpora by construction.

Translationese

IndicMMLU-Pro, Global-MMLU and the machine-translated HLE variants measure translation transfer, not native competence or Indian grounding.

Closed

The one frontier-positioned effort, OpenAI's IndQA, is closed-data and OpenAI-graded — you can't run it, audit its keys, or build on it.

The thesis: India is a multimodal evaluation problem

Text-only evaluation structurally under-measures what matters in India. Four capabilities are core, not accessories:

Voice, both ways

Speech is the primary interface for hundreds of millions. ASR still runs 20–30% WER on Indian languages; STT and TTS/spoken-interaction need frontier evaluation.

Vision, both ways

Understanding India in images and generating historically/contextually accurate images of India.

Language depth

Native-script and code-mixed (Hinglish/Tanglish) competence at frontier difficulty — never translated.

Historical accuracy

A scored dimension throughout — text claims, image content, spoken answers.

What makes it different

Open

Data, harness, and the generation pipeline are public. Anyone can run it, audit answer keys, and reproduce it.

Lab-neutral

Adversarially filtered against all frontier labs' models — not one lab's — with dual judges from two different labs.

Renewable

The pipeline regenerates test waves every 6 months; a private held-out split detects overfitting. It stays hard.

Gap analysis — the most-cited failures (2025–26 surveys)

Findings

  • Translationese ≠ native competence
  • Exam-scrape contamination is endemic
  • Saturation — nothing Indian is frontier-hard
  • Shallow cultural grounding
  • Romanized / code-mixed text ignored
  • Near-zero agentic evaluation in Indic languages

  • Native in-script authoring, never translation (D-009)
  • Synthetic + source-grounded; exams as syllabi only (D-017)
  • Cross-lab adversarial filter keeps only what models fail (D-010)
  • Domain quotas + expert spot-check
  • Hinglish + Tanglish as first-class tracks (D-031)
  • Phase 4 tau2-style Indian agent domains
  • Canary + private split + 6-monthly regeneration (D-013)

One Python package, one schema family, one pipeline, one harness. The pipeline is the product — its output is a dataset, but its openness is the differentiator.

12 language tracks & 10 domains

Tracks: Hindi, Bengali, Tamil, Telugu, Marathi, Kannada, Malayalam, Gujarati, Punjabi, Odia, English-about-India, code-mixed (Hinglish + Tanglish).

Domains: law & constitution · governance & administration · STEM-in-India · medicine incl. Ayush · finance & taxation · agriculture · history & heritage · literature & arts · geography & environment · society & daily life.

Schema extends to all 22 scheduled languages via refresh waves (D-017).

The trust model

Contamination defense

  • Canary GUID in every public file — trainers can filter it out
  • Private held-out split (~20% of public, ~500 items) — never published, detects overfitting
  • Renewable — pipeline regenerates ~1/3 of items every 6 months around a stable anchor subset

Answer-key integrity

  • Keys derived from source documents, not model memory
  • S2 — independent model panel re-derives every key from the source
  • S4 — expert spot-check; 100% of safety-tagged items
  • Release gates block shipping without S0 sources + S4 verdicts

Evaluation harness

Inspect AI (primary)

Clones the inspect_evals/hle pattern: Explanation/Answer/Confidence prompt, model-graded equivalence, accuracy + calibration error. Native support for every provider and for the future agentic phase.

Standalone scripts

predict.py + judge.py — zero-framework, resumable, run locally on Ubuntu against any OpenAI-compatible endpoint (incl. local vLLM/Ollama). Reports accuracy ± 95% CI and RMS calibration error.

Provider abstraction

providers.py gives one complete(model_id, …) call across OpenAI, Anthropic, Google and Sarvam-compatible endpoints (keys from environment). judging.py implements the dual-judge + third-judge-tiebreak protocol with a normalization short-circuit for exact matches (cost control). Empty model responses raise — they are never silently graded as failures, which would corrupt filter yield.

Each stage is a discard gate. Only items that survive all five reach a release.

S3 adversarial filter — the survival rule

Evaluation flow (scoring a model)

Scores compare only within a dated version; refresh waves keep the benchmark ahead of contamination.

How to Use

Install

git clone https://github.com/pukrvi/indibench
cd indibench
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev,eval]"        # add ",pipeline" for the filter run
pytest tests/ -q                    # 16 tests, all offline

Assemble / re-validate the candidate pool (no keys)

python scripts/assemble_candidates.py
# → validates all 240 candidates, writes canary-wrapped files to data/candidates/v0-seed/

Run the filter when API keys arrive

This is the keys-day entry point. Export OPENAI_API_KEY, ANTHROPIC_API_KEY, GOOGLE_API_KEY, SARVAM_API_KEY (+ SARVAM_BASE_URL), pin exact model versions, and smoke-test first.
# 1. tiny smoke run
python scripts/run_filter.py \
  --panel openai/<id> anthropic/<id> google/<id> sarvam/<id> \
  --judges anthropic/<id> google/<id> --tiebreak openai/<id> \
  --limit 10

# 2. full pool (resumes automatically; errored items retry)
python scripts/run_filter.py --panel … --judges … --tiebreak …

# 3. promote survivors into a versioned release (GATED)
python scripts/run_filter.py … \
  --promote-to data/releases --version v2026.XX \
  --sources data/sources/<manifest>.jsonl \
  --audit-verdicts <s4-verdicts>.jsonl

Release is gated: promotion refuses without a registered S0 source manifest and recorded S4 audit verdicts (matching the trust model). See docs/RUNBOOK.md for the exact, current flags.

Offline dry-run (no keys, deterministic mock provider) to exercise the whole chain: python scripts/run_filter.py --mock --skip-s2 --limit 30

Evaluate a model against a release

# Inspect AI (handles every provider natively)
inspect eval evals/inspect/indibench_text.py \
  -T data_file=data/releases/public/v2026.XX/indibench_text_hi_v2026.XX.json \
  --model <provider/model>

# Standalone (resumable; works with local vLLM/Ollama via --base-url)
python evals/standalone/predict.py --data <file> --model <id>
python evals/standalone/judge.py  --data <file> --predictions <out> --judge <id>

Contribute

Report an error

Wrong key, ambiguous question, bad transliteration → open a Question error report issue. Reporters are credited in release notes.

Contribute sources

Hard, well-licensed Indian documents (textbooks, gazettes, case law). Direct question submissions are declined by design — contribute grounding, run the open pipeline.

Lend expertise

Volunteer domain reviewers who spot-check items earn co-authorship consideration on the paper.

Full operational detail — guardrails, resume behavior, private-split handling, model pinning — lives in docs/RUNBOOK.md.
